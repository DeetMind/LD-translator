Technical FAQ

Background on methodology, data requirements, and current limitations.

1. Where are your input data coming from, and what makes them defensible?

Right now this is a calculation engine only, using dummy data. Defensibility would come from whatever event-loss table the user provides. In practice it should come from a licensed cat model (RMS, AIR) or validated public model (FEMA, USACE). I'd welcome input on which models to ingest for which interventions and assets.

2. How are you defining and parameterizing flood mitigation?

Generic scalar parameters (a depth threshold, two reduction percentages). No hydraulic routing, no intervention-specific fragility. A levee and a detention basin use identical parameter slots. Scalars with intervention-specific performance curves derived from hydraulic model runs, feeding into a depth-damage function, would be an improvement.

3. Is this asset level, community level, or translating between the two?

The tool is agnostic and has no translation mechanism. Partial protection (e.g. a levee covering 60% of exposed assets) can't be represented. I can add spatial disaggregation logic and a protection footprint layer so outputs correctly reflect which assets are inside vs. outside the protection zone.

4. What vulnerability functions or damage curves are you using?

None. Gross loss arrives pre-computed; the tool never touches hazard-to-loss conversion. The next version can integrate depth-damage functions (e.g., HAZUS curves) so the tool can derive loss from hazard directly, rather than depending entirely on upstream assumptions it can't inspect.

5. Can you back-test against actual claims data?

No. Outputs are internally consistent but empirically ungrounded. I'm thinking to benchmark mitigated EAL reductions against FEMA BCA toolkit results or USACE study outcomes for comparable intervention types.

6. Is this single project or scalable to portfolios?

Single project only. The data model (one CSV, one mitigation profile) has no aggregation logic, spatial correlation, or multi-intervention support. The data model can be restructured to handle location-level event sets with a protection footprint layer and correlated aggregate losses.

7. What is the development path to real decision value?

I want to close three gaps: (1) a governed data ingestion path from a validated model, (2) actuarially reviewed mitigation parameters, (3) empirical validation against loss experience. Until then this can be a sensitivity analysis and EP curve communication tool, not a pricing input.

8. Cost constraints — can it scale cheaply?

The engine itself would be essentially free, or very low cost for municipal users. The cost problem is upstream inputs. I'm thinking to wire FEMA flood frequency data and HAZUS damage functions as a free, defensible default input source — making the full pipeline viable for small municipalities or parametric products where a full cat model run isn't justified.
